## 2. 25-04 토픽 기기 간 동기화 실패 문제 해결 (SQL 쿼리 오류 수정)
* **원인**: 
  * PC에서 푼 데이터를 핸드폰으로 새로고침할 때, 핸드폰의 로컬스토리지에 기존 세션 ID가 없으면 백엔드 서버가 DB(`app_session` 테이블)에서 가장 최신 세션을 자동 연동해주는 폴백 로직이 실행됩니다.
  * 하지만 해당 SQL 쿼리가 `ORDER BY id DESC` 형식으로 되어 있었으나, 실제 `app_session` 테이블 스키마에는 `id` 컬럼이 존재하지 않아 SQLite에서 쿼리가 완전히 크래시(실패)하는 현상이 있었습니다.
* **해결**:
  * `server/index.js` 내의 두 군데의 세션 조회 쿼리 정규식을 `ORDER BY id DESC` -> **`ORDER BY updated_at DESC`**로 변경하여, 쿼리 크래시 없이 실제 업데이트 일자 기준으로 가장 최신의 PC 풀이 데이터를 핸드폰으로 완벽하게 반환하도록 정정했습니다.

## 3. 락스크린 퀴즈 후보군 전체 확장 (해결방안 A 고도화)
* **해결**:
  1. `slice(0, topicLimit)` 처럼 일부 토픽만 골라내던 필터링을 **완전히 제거**하고, 데이터베이스에 저장된 **전체 토픽**을 후보군으로 삼아 LLM에 함께 전달하도록 로직을 변경했습니다.
  2. 토픽 본문 텍스트 로딩 시 기존의 동기식 `for...of` 루프 대신 **`Promise.all` 비동기 병렬 처리** 방식으로 교체하여, 전체 토픽 로드 시에도 수밀리초 내에 응답이 완료되도록 획기적으로 최적화했습니다.
  3. 최근 출제 질문 제외 목록(`recentQuestions` 캐시) 및 높은 변동성 설정(`temperature: 0.9`)과 완벽히 상호작용하여 다양한 문항이 전 범위에서 골고루 안배되어 출제됩니다.

## 5. 락스크린 문제 생성 실패 및 타임아웃 문제 해결 (API & 지침 맵핑 수정)
* **원인**: 
  1. **지침 누락**: `generateDailyLockscreenQuestions` 함수의 인자로 `lockscreenInstructionsPrompt`가 전달되었으나, 정작 프롬프트 생성용 템플릿(System/User Prompt)에는 해당 변수가 전혀 주입되지 않고 누락되어 있어 AI가 사용자 맞춤형 락스크린 지침을 읽지 못하고 오작동하는 심각한 논리 오류가 있었습니다.
  2. **Vercel 타임아웃 (504)**: Vercel 서버리스 환경은 **10초**의 매우 타이트한 실행 한도(Hobby plan)를 가집니다. 하지만 기존 `callLLMWithFailover` 로직은 Primary Key 실패 시, 불필요하게 429 오류에 대해 **지수 백오프 대기(sleep delay)**를 수행하고, 존재하지 않거나 과부하 상태인 12개 이상의 백업 모델들(`gemini-2.5-flash`, `gemini-3.5-flash` 등)을 연달아 순회하여 시간이 초과되는 근본적인 병목이 있었습니다.
* **해결**:
  1. **프롬프트 맵핑 정상화**: `systemInstruction` 템플릿 하단에 `[출제 지침 기준 (Lockscreen Generation Standards)]` 헤더와 함께 `${lockscreenInstructionsPrompt}` 변수를 정상적으로 주입 및 출력되도록 수정 완료했습니다.
  2. **Vercel 최적화 페일오버 설계**:
     * `isVercel` 환경 변수 감지 시, 429 할당량 초과 에러에 대해 **지수 백오프 대기 시간(sleep)을 0초로 스킵하고 즉시 다음 키/모델로 페일오버**하도록 수정하여 API 응답 속도를 획기적으로 개선했습니다.
     * `executionList` 생성 시, 루프를 키(Key) 기준으로 먼저 그룹화하여, preferred model인 `gemini-3.1-flash-lite`와 검증된 실 운영용 `gemini-1.5-flash`를 핵심 백업으로 순차적으로 우선 배치하였습니다. 불필요하고 실패율이 높은 백업 모델 탐색 단계를 완전히 줄임으로써 10초 제한 시간 내에 안정적으로 퀴즈를 반환하도록 조치했습니다.

## 6. AI 주제 개요 생성 ('개' 버튼) 및 저장/관리 기능 추가 (2026-06-30 패치)

* **배경 & 목적**:
  * 사용자가 학습 중에 특정 토픽에 대해 전체적인 맥락과 핵심 요약을 원할 때, AI 튜터가 구조화된 고품질 개요 및 시험 대비 가이드를 한 번의 클릭으로 즉시 작성해 주도록 기능을 신설했습니다.
  
* **상세 구현 사항**:
  1. **백엔드 (Vite Server)**:
     * `server/plugins/overviewsPlugin.js` 신설: AI 튜터에게 주제별 핵심 개념, 중요 공식/공식 유도, 주요 암기사항, 그리고 전공 시험 대비 꿀팁을 LaTeX와 Markdown 혼용 형식으로 작성하도록 하는 전용 프롬프트 생성 로직 탑재.
     * `GET /api/session/overviews` 및 `POST /api/session/overviews` API 엔드포인트 개설: 사용자별 커스텀 개요 리스트를 SQLite DB (`app_session` 테이블) 내 `formula_overviews` 레코드로 영속 보존 및 실시간 동기화.
     * `/api/chat` 라우트에 `overviewMode` 감지 로직 및 전용 핸들러 바인딩.
  2. **프론트엔드 (Vite Client)**:
     * **네 군데의 AI 도구 버튼 바 확장**: PC/모바일 학습 화면 및 주관식 퀴즈 뷰의 기존 "두" (두문자) 버튼 왼쪽에 붉은 장미색 테마의 **"개" (개요) 버튼** 추가.
     * **주제 개요 팝업 (Overview Prompt Modal)**: "개" 버튼 클릭 시 입력 창이 열리고, 원하는 임의의 주제나 키워드를 입력해 AI 생성 요청 가능.
     * **필수공식 리스트 탭 확장**: "공식", "표", "앞글자" 탭 옆에 **"개요"** sub-tab 추가.
     * **개요 데이터 렌더러**: `LatexRenderer`를 완벽히 연동하여 AI가 도출한 개요 마크다운과 복잡한 수학 공식(LaTeX)이 고해상도 수식 컴포넌트로 자동 렌더링되도록 구현.
     * **상태 관리 & Pull-to-refresh**: 개요 새로고침 API, 개요 카드 접기/열기 상태 보존, 더블클릭을 통한 개요 제목(Title) 수정 기능, 개별 카드 삭제 기능 추가 및 모바일 풀투리프레시(Pull-to-refresh) 리스트에 연동.

## 7. 믹스 복습(오늘의 필수 표/앞글자 믹스 복습) AI 재출제 먹통 오류 수정 (2026-06-30 패치)

* **원인**:
  * 믹스 복습(`mixed_acronym_table`)은 실제 데이터베이스의 `topics` 테이블에 행(row)이 존재하는 일반 토픽이 아니라, 사용자가 보관한 표와 앞글자를 조합하여 로컬에서 동적 생성하는 가상 토픽입니다.
  * 기존 `handleRefreshReviewQuestions` (AI 재출제 버튼의 클릭 핸들러)는 이를 구분하지 않고 서버의 `/api/topics/mixed_acronym_table_schedule/ai-questions` API로 재생성 요청을 무조건 전달했습니다.
  * 서버에서는 존재하지 않는 가상 토픽 ID에 대해 404/500 에러를 반환하였고, 프론트엔드는 이 에러로 인해 응답 대기 상태(로딩 상태)에서 탈출하지 못하고 먹통(Freeze)되는 현상이 발생했습니다.

* **해결**:
  * `client/src/App.jsx` 내 `handleRefreshReviewQuestions` 함수 진입부에 `mixed_acronym_table`에 대한 예외 처리 분기 로직을 구현했습니다.
  * 믹스 복습 토픽인 경우, 서버의 AI 질문 생성 API 대신 `handleOpenAIQuestions`와 동일한 방식으로 로컬 데이터베이스의 표/앞글자 풀을 불러옵니다.
  * 단, 재생성 목적에 부합하도록 매번 **새로운 무작위 시드(Random Seed)**로 셔플을 유도하여 사용자가 클릭할 때마다 5개의 복습 세트가 무작위로 새로 채워지며, 기존 입력/답안/세션 정보를 완전 초기화한 뒤 서버 세션에 다시 등록하도록 설계하여 원천적인 먹통 현상을 해결했습니다.

## 8. 표 내보내기 제목 파서 개선, 보관함 표 행 삭제 기능 추가 및 화면 먹통 버그 수정 (2026-06-30 패치)

* **상세 구현 사항**:
  1. **표 내보내기 제목 자동 정제 (markdownTableRenderer.js)**:
     * **문제**: 마크다운 본문에서 표 바로 윗줄을 제목 후보로 삼을 때, 줄 전체가 긴 설명글(예: `**주파수별 기준**: 진동의 주파수(Hz)가 낮을수록...`)인 경우 전체 문장이 표 제목으로 들어가 복잡해지는 문제가 있었습니다.
     * **개선**: 제목 후보 줄에 콜론(`:` 또는 `：`)이 포함되어 있고 그 앞의 텍스트가 40자 이하의 적합한 제목 후보이면 콜론 앞부분만 발라내고 markdown 볼드 기호(`*`)를 제거한 깔끔한 타이틀(예: `주파수별 기준`)만 추출하도록 개선했습니다. 또한, 전체 제목 길이를 최대 40자로 제한했습니다.
  2. **보관함 표 행 개별 삭제 기능 추가 (App.jsx)**:
     * **문제**: 저장된 비교표의 특정 행(Row)만 필요 없어져서 지우고 싶어도 개별 행 수정이 불가능했습니다.
     * **해결**: "표" 보관함 탭의 렌더링 방식을 기존 정적 HTML(`dangerouslySetInnerHTML`) 주입 방식에서, `parseHtmlTable`로 파싱하여 리액트 컴포넌트로 렌더링하는 동적 구조로 개편했습니다. 
     * 표의 가장 우측에 **[삭제]** 열을 신설하고 각 행마다 휴지통(Trash2) 버튼을 부착하여, 해당 행을 클릭해 손쉽게 삭제할 수 있게 했습니다. 삭제 시 남은 행 정보를 HTML 형식으로 실시간 재조립(`rebuildTableHtml`)한 후 데이터베이스에 즉시 동기화해 영속 저장합니다.
  3. **화면 먹통 오류 해결 (expandedOverviewIds 정의 누락 수정)**:
     * **원인**: 개요 탭을 렌더링할 때 개요 카드의 열고 닫힘 상태를 제어하는 `expandedOverviewIds` 상태 변수의 React `useState` 선언이 누락되어 `ReferenceError`가 발생해 화면 렌더링 전체가 크래시 및 먹통이 되었던 것이었습니다.
     * **해결**: `client/src/App.jsx`에 `const [expandedOverviewIds, setExpandedOverviewIds] = useState({});` 상태 변수를 추가 선언하여 크래시와 화면 먹통 현상을 완벽히 해결했습니다.

## 9. 실시간 튜터 대화 맥락 기반 AI 표 정제/작성 기능 추가, 개요 마크다운 표 구성 및 학습 팁 영역 제거 (2026-06-30 패치)

* **상세 구현 사항**:
  1. **실시간 튜터 대화 맥락 기반 AI 표 정제/작성 기능 (suggest-title-and-refine)**:
     * **문제/요구**: 실시간 튜터 창에서 공부하다가 생성된 적당한 표를 내보낼 때, 대화 맥락을 감안한 맞춤형 AI 작성이 요구되었습니다.
     * **해결**: 
       - 백엔드에 `/api/table/suggest-title-and-refine` POST API 엔드포인트를 신설하여 표 원본 HTML과 실시간 튜터 대화 맥락(최근 대화 기록)을 분석하도록 구축했습니다.
       - AI가 대화 주제에 걸맞은 가장 적절하고 전문적인 표준 표 제목(Title)을 추출하고, 내용에 기술사 표준 용어와 오탈자 수정을 가해 정제된 `<table>` HTML 구조로 자동 재작성합니다.
       - 프론트엔드에서 내보내기 버튼 클릭 시 로딩 스피너 및 AI 분석 인디케이터가 포함된 프리미엄 모달을 노출하며, AI 분석 결과를 표 제목 입력칸과 내용에 즉시 자동 완성해 줍니다.
  2. **개요 탭 구조 변경 (overviewsPlugin.js)**:
     * **요구**: 개요 내용을 마크다운 표(1열: 구분 [개요(40~60자), 메커니즘, 직관적 의미], 2열: 내용, 총 4개행) 형태로 구성합니다.
     * **해결**: 
       - `overviewsPlugin.js` 내 기본 개요 데이터를 상기 규격(구분/내용 2열 구성, 개요/메커니즘/직관적의미 4개행)의 마크다운 표 문자열로 전면 업데이트했습니다.

## 14. 비교표 내 '비고' 열 행 삭제 버튼의 onclick 속성 내 $ 문자 이스케이프 유실로 인한 테이블 렌더링 붕괴 버그 해결 (2026-07-07 패치)

* **원인**:
  * 최근 패치(`1e8acb2`)에서 공식 개요 비교표의 행을 개별적으로 삭제할 수 있도록 각 행의 끝에 "비고(행 삭제)" 열과 휴지통 버튼을 생성하는 기능이 도입되었습니다.
  * 이때 삭제 버튼의 인라인 `onclick` 속성에 해당 테이블의 전체 마크다운 문자열(`entireTableEscaped`)을 문자열 인자값으로 전달하는데, 표 내용 중 수식 표현용 달러 기호(`$`)가 그대로 노출되어 있었습니다.
  * 테이블 HTML이 마크다운 렌더러를 거친 후 `LatexRenderer` 컴포넌트로 전달되면, `LatexRenderer`는 정규식(`/\$([^\$\n<>]+?)\$/g`)을 이용해 텍스트 내 수식(예: `$B = 1$`, `$A = 1$`)을 찾아내어 KaTeX HTML(내부에 `class="katex"` 같은 이중 따옴표가 포함된 태그)로 치환합니다.
  * 이 치환 작업이 버튼 태그 내부의 `onclick="..."` 속성 문자열 안에서도 무차별적으로 실행되면서, KaTeX HTML의 이중 따옴표가 `onclick="..."` 전체를 감싸는 이중 따옴표를 중간에 닫아버리게 만드는 문법 충돌이 발생했습니다.
  * 결국 HTML 속성이 깨지면서 삭제 대상이 아닌 수식 문자열과 자바스크립트 소스 코드 일부(`') } else { alert(...)`)가 화면 상의 테이블 셀 내부 텍스트로 고스란히 튀어나오는 심각한 비주얼 및 데이터 파싱 붕괴를 초래하였습니다.

* **해결**:
  * `client/src/utils/markdownTableRenderer.js` 파일에서 `onclick` 속성에 인자로 들어갈 변수들(`firstCellSafe`, `cleanTitleSafe`, `entireTableEscaped`)을 인라인 조립할 때, 달러 기호(`$`)를 HTML 안전 엔티티 코드인 **`&#36;`**로 치환(`replace(/\$/g, '&#36;')`)하도록 전면 보강했습니다.
  * `&#36;`는 HTML 문자열 단계에서 `$` 기호로 취급되지 않으므로 `LatexRenderer`의 수식 변환 필터링을 완벽하게 우회하게 되며, 이후 브라우저가 HTML을 로드하여 DOM을 빌드하고 `onclick` 이벤트 리스너를 실행하는 시점에는 자동으로 `$`로 원상 복구(디코딩)되어 동작합니다.
  * 수정 완료 후 Vite 프로덕션 빌드를 다시 진행하여 컴파일 정상 여부를 점검하였고, 배포 리포지토리(`origin main`)로 커밋 및 푸시하여 실시간 반영 처리를 마쳤습니다.

* **원인**:
  1. **개행 구분자 유실**: AI가 생성한 개요 비교표 마크다운은 줄바꿈이 일반 개행(`\n`)이 아닌 `<br>` 태그 형태로 저장 및 관리됩니다. 하지만 기존 `window.__handleGlobalRowDelete` 함수는 줄바꿈을 단순 `\n`으로 스플릿하여 행들을 나눈 뒤 일반 `\n`으로 다시 합쳐서(`join('\n')`) 업데이트했기 때문에, 데이터베이스에 저장할 때 표 레이아웃 형식이 무너져 렌더링되지 않는 오류가 있었습니다.
  2. **매칭 탐색 버그**: HTML 상에서 `onclick` 리스너를 호출할 때 전송되는 테이블 마크다운은 이스케이프/인코딩된 달러 임시 문자(`__DOLLAR_TEMP__`) 및 문자열 내 줄바꿈 이스케이프(`\n` -> `\\n`) 상태였습니다. 이를 온전히 언이스케이프하지 않아 기존의 `contentStr.includes(originalTableMd.trim())` 패턴 탐색에서 대상 비교표 영역을 찾지 못하여 데이터베이스 영구 저장이 누락되고 임시 삭제 안내 모달만 반복 표출되었습니다.
  3. **재생성 데이터 무결성**: React 렌더링 트리 상의 개별 삭제 핸들러에서도 `rebuildMarkdownTable(headers, updatedRows)`를 호출 시 구분자 파라미터가 생략되어 일반 개행 `\n`으로 마크다운이 재조립되어 표 깨짐이 동일하게 누적되고 있었습니다.

* **해결**:
  1. **임시 달러 및 개행 복원 적용**: `__handleGlobalRowDelete`에 전달받는 인자들에 대해 `__DOLLAR_TEMP__`를 다시 `$`로, `\\n`을 실제 개행 `\n`으로 온전히 치환 및 복원하도록 정규식을 개선했습니다.
  2. **비교표 셀 분할 및 `<br>` 재조합 최적화**: 
     - 표 마크다운 셀을 파싱할 때 좌우의 여백을 포함하여 파싱 오류가 없도록 셀 양단 정규화 및 트림을 고도화했습니다.
     - 재생성되는 마크다운 테이블은 구분자로 `<br>`을 사용하도록 `rebuildMarkdownTable`에 `separator` 매개변수를 신설하여, 행들을 정상적으로 `<br>` 태그로 조인하게끔 조치했습니다.
  3. **비교표 정규화 비교 탐색 탑재**:
     - 개요 본문 내에 표가 존재할 때 HTML 개행과 마크다운 개행의 차이로 문자열 비교가 빗나가지 않도록, `<br>` 태그와 공백을 일률적으로 정규화해주는 `normalizeTable` 헬퍼 함수를 추가 적용했습니다.
     - 정규화된 텍스트 비교를 통해 타겟 표를 정확하게 탐색 및 교체하도록 매칭 로직을 개편했습니다.
  4. **비교표 파싱 시 다중 파이프(||) 및 공백 누적으로 인한 테이블 깨짐 및 열 치우침 오류 해결 (2026-07-07 2차 패치)**:
     - **원인**: 개요 전체 테이블 내에 중첩된 비교표를 셀 데이터로 저장하고 불러오는 과정에서, 양 끝에 `| |` 형태로 중복 파이프라인 및 빈 필드가 누적 생성될 수 있었습니다. 이 상태에서 행을 분할(`split('|')`)하고 좌우 하나씩만 잘라내는 방식(`slice(1, -1)`)을 쓰면, 앞뒤의 빈 문자열(`""`) 컬럼이 정형 데이터 헤더 목록(`headers`)에 포함되어 실제 열 개수(예: 3개)보다 파싱 결과(예: 5개)가 더 많아져 열이 밀리고 빈 열이 생기는 불균형이 발생했습니다.
     - **해결**: 마크다운 행 파서인 `parseRow`와 `parseMarkdownTable` 내에 양 끝에 연속으로 나타나는 빈 셀 문자열을 전부 루프를 돌며 잘라내고 핵심 열만 반환하는 `while` 기반의 정형화 알고리즘(`parseRowCells`)을 구축했습니다. 이를 통해 DB에 비정상적인 파이프가 껴 있더라도 화면 로딩 및 렌더링 시점에 즉시 동적으로 교정하여 열이 밀리지 않고 정상 레이아웃으로 출력되도록 수정했습니다.
  5. **비교표 치환 및 파싱 시 닫는 파이프(|) 유실로 인한 후속 행(예: 주요 한계 등) 누락 오류 해결 (2026-07-07 3차 패치)**:
     - **원인**: 개요 비교표의 맨 마지막 행에 더블 파이프(`| |`)가 저장된 상태에서 데이터베이스 로드 시점의 `healedContent.replace(/\|[ \t]*\|/g, '\n|')` 로직이 이를 감지하여 맨 끝에 인위적으로 `\n|` 개행을 생성하면서 마지막 행의 닫는 파이프라인(`|`) 기호가 유실되는 현상이 있었습니다. 이로 인해 마크다운 테이블 검증 함수 `parseMarkdownTable` 내의 `line.endsWith('|')` 검사가 실패하여 테이블 파싱이 조기에 중단되었고, 그 결과 마지막 데이터 행(`주요 한계` 등)이 화면에 렌더링되지 못하고 누락되는 현상이 발생했습니다.
     - **해결**: `parseMarkdownTable` 내의 행 검출 정규식 및 유효성 조건을 개편하여 `line.startsWith('|') && line.endsWith('|')` 대신 `line.startsWith('|') && line !== '|'`로 완화함으로써, 백엔드 개행/파이프 치환 로직에 의해 닫는 파이프가 유실된 행이라도 테이블의 데이터 행으로 안전하게 포함하여 온전히 전체 비교표가 다 렌더링되도록 수정했습니다.
  6. **핸드폰 세로보기 시 비교표 데이터 열 100% 화면 채움 및 오직 비고(삭제) 열만 우측 화면 밖 배치 최적화 (2026-07-07 4차 패치)**:
     - **원인**: 모바일 세로보기(360px~430px) 환경에서 비교표가 가로 화면에 다 들어가려다 보니 셀 너비가 좁아지고 가독성이 저하되는 문제가 발생했습니다. 이전 패치에서 `min-w-[650px]`로 단순 최소폭을 늘렸더니, 주요 내용 열마저 일부 잘려서 처음 로딩 시점에 가로 화면에 완전히 들어오지 않는 불편함이 발생했습니다.
     - **해결**: 모바일 화면 크기에 정밀 연동되도록 테이블 가로 너비를 `w-[calc(100%+64px)] min-w-[calc(100%+64px)]`로 변경했습니다. 또한 `비고` 컬럼 및 셀의 크기를 `w-16 min-w-[64px]` (64px)로 고정했습니다. 이를 통해 최초 화면 진입 시 **핵심 비교 내용 열(`구분`, `간편법`, `상세해석법` 등)은 모바일 가로 100% 화면에 꽉 채워서 완벽히 출력**되고, **오직 `비고` 삭제 열만 정확하게 화면 오른쪽 밖(64px 오버플로우)으로 정렬**되어, 손가락으로 가볍게 왼쪽으로 밀어 삭제 버튼을 노출하도록 UX를 정밀 튜닝했습니다. (데스크톱에서는 원래대로 `w-full`로 보여줌)
  7. **복습/학습 세션에서 주관식(특히 계산/그림문항) 답안 입력 창이 이중으로 중복 노출되는 오류 해결 (2026-07-07 5차 패치)**:
     - **원인**: 복습 문항을 그리는 카드 컴포넌트 내에 `isSubj && !(q.tableData || q.comparisonTableData)` 조건으로 주관식 단답형 입력을 그리는 독자적인 블록(Block 1)과, 뒤이은 `isSubj` 조건 하에 각 유형별 분기를 처리하는 4단 중첩 삼항 연산자 블록(Block 2)이 공존하고 있었습니다. 이 중 Block 2의 3번째 분기(Ternary 3)가 표채우기/서술형이 아닌 일반 단답형/계산형 주관식을 렌더링하도록 예외 처리 없이 똑같은 입력 필드를 렌더링하여, 일반 주관식 단답(그림/계산 문항 포함) 렌더링 시 두 블록 모두에 매칭되어 화면에 동일한 답안 입력 창과 제출 버튼이 연달아 이중 표출되는 원인이 되었습니다.
     - **해결**: 중첩 삼항 연산자(Block 2) 내의 3번째 분기(단답/계산 등)가 이미 모든 유형별 조건 분기를 엄격하게 충족하여 렌더링하고 있으므로, 상단에 독립적으로 존재하며 전반적인 예외 처리가 부족해 중복 노출을 유발하던 레거시 블록(Block 1, lines 20282-20417)을 완전히 삭제 처리하여 모든 주관식 문항이 유형에 관계없이 단 한 번만 정확한 레이아웃으로 렌더링되도록 교정했습니다.
  8. **비교표/장단점 마크다운 표의 열 헤더 클릭 편집 기능 구현 (2026-07-07 6차 패치)**:
     - **요구사항**: 개요 영역의 비교표/장단점 마크다운 테이블에서도 공식 테이블 탭에 구현되어 있는 방식과 동일하게 열 헤더(구분, 컬럼명 등)를 클릭하여 즉시 수정하고 서버에 동기화할 수 있도록 지원을 요청했습니다.
     - **해결**: 개요 비교표의 `<thead>` 헤더 목록 렌더링에 `activeEditCell` 상태 값을 연동했습니다. 특정 열 헤더 셀을 클릭하면, 해당 셀이 `<textarea>` 입력 창으로 전환되며 즉시 텍스트를 편집할 수 있습니다. 입력 후 포커스를 해제(`onBlur`)하거나 Shift 없이 Enter 키를 입력하면, 수정된 헤더 값으로 마크다운 테이블 구조를 재생성(`rebuildMarkdownTable`)하여 개요 콘텐츠(`ov.content`)를 안전하게 치환 후 클라우드 서버에 자동 동기화(저장)하도록 구현했습니다.
  9. **믹스 복습 내 비교표/장단점 채우기 퀴즈 테이블 열 헤더 드래그 조절 기능 구현 (2026-07-07 7차 패치)**:
     - **요구사항**: 믹스 복습 및 학습 세션 중 출제되는 비교표/장단점 채우기 주관식 퀴즈 테이블에서, 모바일 세로보기 환경에서 열 너비를 드래그하여 임의로 늘리거나 줄일 수 있도록 조절(Resizing) 기능 구현을 요청했습니다.
     - **해결**: 주관식 퀴즈 테이블 컴포넌트(`TableQuiz`) 내부의 비교표 테이블(`compTable`) 영역에 독자적인 열 조절 상태(`compColWidths`, `compMobileColWidths`) 및 드래그 핸들 이벤트 리스너(`startCompColumnResize`)를 구축했습니다. 이제 모바일 세로보기 화면에서 비교표 퀴즈를 풀 때 열 헤더 우측 경계선을 터치/드래그하여 컬럼 너비를 유연하게 늘려 쓸 수 있습니다. 조절된 너비 값은 `localStorage`에 질문 고유 ID별로 로컬 캐싱되어 세션이 유지되며, 첫 번째 열 헤더(`구분`)를 더블 탭하면 기본 크기로 즉시 초기화되도록 조작 편의성을 극대화했습니다.

      - **해결**: `TableQuiz`(공식 테이블/비교표) 및 `AcronymQuiz`(앞글자 표) 내의 모바일 터치 드래그 리사이즈 함수(`doResize`) 구조를 개편했습니다. 1열(`idx === 0`) 조절 시에는 독립적으로 너비가 변경되지만, 2열 이상(`idx >= 1`)의 내용 컬럼의 드래그 핸들을 조작할 경우에는 루프를 통해 1열을 제외한 모든 내용 컬럼들(`i >= 1`)의 너비(`mobileColWidths` / `compMobileColWidths`)를 동일한 조절폭(`newWidth`)으로 동시 업데이트 및 일괄 로컬 캐싱하도록 정밀 튜닝했습니다. 이를 통해 모바일 화면 좁은 공간 내에서 표의 대칭 비율이 깨지지 않고 완벽한 비율로 동시 조절됩니다.
  11. **모바일 세로보기 2열 마크다운 표의 가로 스크롤 제거 및 한 화면(100% 폭) 채움 기능 구현 (2026-07-07 9차 패치)**:

## 24. 표채우기 지문 내 알파벳 범위 기호 중복 및 중첩 치환 버그 해결 (2026-07-11 2차 패치)

* **원인**:
  - 표채우기(Table Quiz) 문항이 출제될 때, 지문에 알파벳 빈칸 범위를 나타내는 물결 기호(`(A)~(J)`) 혹은 한글 단어(`(A)부터 (J)`) 형식의 범위 문자열이 존재하면, 기존의 `multiPattern` 정규식(`/\([A-Z]\)(?:\s*(?:,|\s+및|\s+또는|와|과)\s*\([A-Z]\))+/g`)이 이를 감지하지 못하고 스킵(skip)했습니다.
  - 이로 인해 후속 예외 처리인 단일 빈칸 매칭 `singlePattern` (`/\([A-Z]\)/g`)에 걸리게 되며, 지문 내에 포함된 `(A)` 와 `(J)` 기호 각각이 전체 입력 리스트(A~J 전체 목록)로 개별 치환되는 심각한 결함이 발생했습니다. 결과적으로 `(A), (B), ... (J) ~ (A), (B), ... (J)` 와 같이 내용이 이중으로 중복 출력되는 현상이 발생했습니다.

* **해결**:
  1. **정규식 치환 범위 감지 기능 확장 ([latexUtils.js](file:///c:/Users/airfo/OneDrive/바탕 화면/안티/server/utils/latexUtils.js))**:
     - 다중 빈칸 감지용 `multiPattern` 정규식의 구분자 종류에 범위 표현식들(`~`, `-`, `부터`, `에서`)을 추가 반영하여 `/\([A-Z]\)(?:\s*(?:,|\s+및|\s+또는|와|과|~|-|부터|에서)\s*\([A-Z]\))+/g`로 전면 고도화했습니다.

       2. **[기본 지침 삭제 버그]**: Vercel 프로덕션 환경의 파일 시스템은 읽기 전용(Read-Only)이므로, 사용자가 브라우저 UI에서 기본 지침을 삭제하더라도 서버 파일(`generationStandards.js` 등)의 내용을 물리적으로 바꿀 수 없습니다. 이에 따라 DB가 최신(dbIsNewer)인 상태임에도 불구하고, `mergeStandards`가 Vercel 코드 번들 내 파일(`fileList`)에 여전히 남아 있는 삭제된 기본 지침을 발견하고 "새로 추가된 파일 지침"으로 오인해 지속적으로 DB에 다시 밀어 넣는(Resurrection) 치명적인 맹점이 있었습니다.
     - **해결**:
       - 서버 기동 시 및 지침 GET API 호출 시, 로컬 파일 시스템 상의 지침 소스 파일 수정 시각(`fs.statSync(filePath).mtime`)과 데이터베이스 캐시의 최종 업데이트 시각(`updated_at`)을 비교하는 `checkIsFileNewer` 함수를 도입하였습니다.
       - 파일 수정 시각이 데이터베이스 캐시보다 최신일 때(개발자가 직접 파일을 수정한 경우)는 파일을 절차적 기준(Source of Truth)으로 삼아, 파일에서 삭제된 지침은 데이터베이스 캐시에서도 온전히 소멸하고 추가된 내용은 DB에 즉시 적재하도록 완벽히 정비했습니다.
       - **[삭제 유지 로직]**: `dbIsNewer`인 상황(사용자가 UI에서 저장/삭제 조치한 경우)에서는 **추가적인 파일 루프 병합을 스킵하고 데이터베이스 목록(`dbList`) 자체를 100% 진정한 기준(Source of Truth)으로 신뢰하고 반환**하도록 동기화 병합 알고리즘을 획기적으로 개선했습니다. 이를 통해 사용자가 UI에서 삭제한 기본 지침이 Vercel의 읽기 전용 파일 데이터에 의해 불필요하게 부활하는 현상을 완전히 차단했습니다.
  7. **지침 GET API 내 catch 블록의 닫는 중괄호 누락 구문 오류(SyntaxError) 수정 ([configRoutes.js](file:///c:/Users/airfo/OneDrive/바탕 화면/안티/server/routes/configRoutes.js))**:
     - **원인**: 이전 `multi_replace_file_content`로 `configRoutes.js` 내의 `checkIsFileNewer` 함수 연동 코드를 치환할 때, `catch (dbErr) { ... }` 블록의 닫는 중괄호(`}`) 기호가 네 군데의 GET 라우트에서 전부 소실되는 치명적인 결함이 발생했습니다. 이로 인해 `SyntaxError: Unexpected token 'catch'`와 함께 Express 백엔드 서버가 로딩 시점에 완전히 크래시하여 모든 API 요청에 500 응답이 반환되고 기기 연결이 끊어지는 장애가 발생했습니다.
     - **해결**: 네 군데의 GET 라우트(`engineering-standards`, `grading-standards`, `generation-standards`, `lockscreen-standards`) 내의 catch 블록 끝에 누락된 닫는 중괄호(`}`)를 정상적으로 보완 및 삽입하여 구문 크래시를 원천 해결했습니다.
  8. **서버 기동 시 지침 목록 변수 미정의 참조 오류(ReferenceError) 해결 ([index.js](file:///c:/Users/airfo/OneDrive/바탕 화면/안티/server/index.js))**:
     - **원인**: `index.js` 내의 `initializeAllStandards`에 파일 수정 시간 기반 동기화 로직을 탑재하면서, 정작 지침들의 원본 배열 객체인 `standardsList`, `gradingStandardsList`, `generationStandardsList`, `lockscreenStandardsList` 파일 임포트 선언이 누락되어 있었습니다. 이로 인해 Node.js 구동 과정에서 `ReferenceError: gradingStandardsList is not defined`가 발생하여 백엔드 기능이 가동 직후 비정상 종료되는(500 에러 및 FUNCTION_INVOCATION_FAILED) 문제가 있었습니다.
     - **해결**: `index.js` 상단에 각 지침 목록을 내포하고 있는 플러그인 소스 파일들로부터 해당 배열 데이터 임포트 문을 온전히 보강 및 연동하여 시동 크래시를 근본 차단했습니다.
  9. **표 채우기 1열 행 구분 항목의 최소/최대 글자수 제약 가이드라인 전면 제거 ([quizRoutes.js](file:///c:/Users/airfo/OneDrive/바탕 화면/안티/server/routes/quizRoutes.js))**:
     - **원인**: 이전 릴리즈에서 지문 내의 `(15~45자)` 문구는 제거했으나, 백엔드 문제 출제용 원본 프롬프트 템플릿(`quizRoutes.js` 내 `[구분 항목(행 제목) 명확화 및 행동 유도 원칙]`) 내부에 `글자수는 반드시 최소 15자에서 최대 45자 이내로 구체적이고 길게 작성하십시오`라는 AI 유도 프롬프트 지시사항이 그대로 남아 있었습니다. 이로 인해 AI가 표를 채울 때 맨 왼쪽 1열의 구분 명칭(행 헤더)을 직관적인 단어 대신 설명형 문장으로 매우 길게 작성하여 표의 레이아웃을 망가뜨리는 문제를 야기했습니다.
     - **해결**: 프롬프트 내 글자 수 제약 및 불필요하게 설명문 형태로 길게 쓰도록 유도하는 지시와 예시들을 전면 소거하고, 사용자가 채워야 할 지식 범주를 지시하는 "단순하고 직관적인 단어 또는 명사형 어구"로 작성하도록 프롬프트를 깔끔하게 변경했습니다.
  10. **통합 컴파일 및 빌드 검증**:
     - 서버 구문 분석(`node --check`) 및 Vite 클라이언트 프로덕션 컴파일(`npm run build`)을 100% 성공 확인 후 원격 리포지토리(`origin main`) 배포 완료했습니다.
